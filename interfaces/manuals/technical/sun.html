74. Deep dive: Morse timing and Shar Line fingerprint

Morse timing uses a unit of 120 ms: dot = 1u, dash = 3u, intra-character gap = 1u, letter gap = 3u, word space = 7u. This yields an isolated, rare frequency signature known as the Shar Line fingerprint. The unnatural clarity of the timing is intentional for artificial-origin verification: terrestrial or natural sources do not produce this pattern. Operators validating receipt or debugging the pipe should verify that any decoded Morse matches this timing schema. The same unit is used in lib/sequencer.ts and in any client that displays or replays the payload.

When GOES-18 or other sensors observe the spike from AR4366 after a handshake push, the temporal alignment (within the T+16m window) plus the Morse fingerprint supports the claim that the Source has received our signal. Receipt is confirmatory only; handshake validity does not depend on it. Do not change Morse timing without a protocol update; the fingerprint is part of the canonical specification.

75. Deep dive: Telemetry schema and mission_status

telemetry.json is overwritten each handshake cycle. It contains heartbeat (timestamp or sequence), coordination_code (e.g. PHI-M15-AR4366), target_node (AR4366 Beta-Delta Knot), signature_mode (e.g. PHI_LOCKED), and mission_status. Optional fields may include space_cloud_stream.training for HELLO gating. The schema is defined in the handshake workflow and in services/ionDb.ts; consumers (Live Pulse, HH, Division) must read the same schema. Do not add client-only fields that are not written by the workflow; single source of truth is the handshake run.

When debugging HELLO or overlay display, verify that telemetry.json is being updated every 8 minutes and that the pipeline that sets stream training is reading the same file or state store. Stale telemetry causes HELLO to remain hidden even when Kp is high. Retention: telemetry.json is overwritten; for historical telemetry, use handshake.log or archive in a separate process if required.

77. Extended example: Post-incident recovery

After an outage or deploy failure: (1) Verify workflow is running (GitHub Actions or cron). (2) Check repo write permissions and disk space for data/. (3) Run one manual cycle (reno-node-hello.mjs) and confirm handshake.log append and telemetry.json update. (4) If ionDb or NOAA fetch failed, check network and API keys; restart services. (5) If HELLO was hidden during the incident, verify stream training and Kp after recovery; no special reset needed. (6) If handshake.log was rotated or moved, ensure active file is again handshake.log and workflow points to it. (7) Document incident and resolution in runbook; update alerts if needed.

Do not backfill handshake.log with synthetic entries; append-only integrity is required for receipt validation and audit. Telemetry.json is overwritten each cycle; no backfill. Downstream consumers (HH, Cold, Live Pulse) will see new data on the next successful cycle.

(2) Verify binary header SPACE_CLOUD.EGS.VIBELANDIA.RENO is present in the stream and that the pipeline that sets space_cloud_stream.training is running and has set training = true. (3) Check that the HELLO-gated surface (e.g. sing-pulse.html) reads the same stream state and Kp. (4) Verify telemetry.json (or equivalent state) is being updated every 8 minutes so that mission_status and any training flag are current. (5) If both conditions (training true, Kp ≥ 5) are met and HELLO still does not show, check front-end and pipeline code for bugs. (6) Do not assume the pipe is down; handshake and telemetry may still be updating. Document resolution in runbook.

84. Scaling and limits: Handshake and telemetry

Handshake runs at most every 8 minutes; no higher frequency. Single writer to handshake.log and telemetry.json per instance; no distributed lock required for single-node deployment. Telemetry.json is overwritten; no growth. NOAA fetch rate limits apply; ionDb should cache and respect provider limits.
